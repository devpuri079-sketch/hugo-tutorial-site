# Hugo概要

## 現状 IT 開発の現場でのドキュメント管理状況

- **Word / Excel / PowerPoint での管理が依然として主流**  
  多くの企業では、仕様書・設計書・手順書が Office ファイルで作成され、ファイルサーバーや SharePoint に保存されている。  
  しかし、バージョン管理が属人的になりやすく、最新版がどれか分からない、レビュー履歴が追いにくいといった課題が残る。

- **ファイルサーバーの階層構造が複雑化しやすい**  
  プロジェクトが増えるほどフォルダ階層が深くなり、目的の資料にたどり着くまで時間がかかる。  
  検索性も低く、情報が “埋もれる” ことが多い。

- **ナレッジ共有が進みにくい**  
  ドキュメントが散在し、プロジェクト横断での知見共有が難しい。  
  結果として、同じ調査や同じミスが繰り返される。

- **開発者視点では Git で管理したいが、非エンジニアとの共有が難しい**  
  Markdown や GitHub を使いたくても、社内全体での運用に乗せるにはハードルがある。

こうした背景から、**「ドキュメントを Web 化して、誰でもアクセスしやすく、更新しやすい形にする」** というニーズが高まっている。

---

## ドキュメントを Web 化するとは？

- **ドキュメントを “Web ページとして公開・閲覧できる形にする” こと**  
  ブラウザで閲覧できるため、OS やアプリに依存しない。  
  スマホ・タブレットでも読める。

- **リンク構造で情報が整理され、検索性が向上する**  
  フォルダ階層ではなく、Web サイトのナビゲーションとして整理できるため、目的の情報にたどり着きやすい。

- **更新履歴や差分管理が容易になる**  
  Git と組み合わせることで、誰が・いつ・何を変更したかが明確になる。

- **社内 Wiki のように使えるが、より柔軟で軽量**  
  Confluence や Notion のような SaaS とは異なり、静的サイトとしてホスティングできるため、  
  セキュリティ要件が厳しい環境でも運用しやすい。

---

## Web 化の方法（Markdown ⇒ SSG）

- **Markdown でドキュメントを書く**  
  - 軽量で読みやすい  
  - Git で管理しやすい  
  - コードブロックや図表も簡単に書ける  
  - レビューしやすい（差分が見える）

- **SSG（Static Site Generator）で Web ページに変換する**  
  SSG は Markdown を HTML に変換し、静的な Web サイトを生成するツール。  
  代表例：Hugo、Jekyll、Next.js（静的出力）、MkDocs、Docusaurus など。

- **SSG を使うメリット**  
  - 高速（静的ファイルなのでサーバー負荷が低い）  
  - セキュア（動的処理がないため攻撃面が少ない）  
  - CI/CD と組み合わせて自動デプロイできる  
  - テーマやレイアウトを統一できる  
  - 長期運用に強い（HTML は将来も読める）

Markdown → GitHub → SSG → HTML → 社内 Web サーバー  
という流れで、ドキュメントが “生きたナレッジ” として循環するようになる。

## Hugo とは

- **Go 言語製の高速な静的サイトジェネレーター（SSG）**  
  世界的に利用者が多く、特に技術ドキュメントやブログ用途で人気。

- **特徴**  
  - とにかくビルドが速い（数千ページでも一瞬）  
  - 設定がシンプルで学習コストが低い  
  - 豊富なテーマがあり、デザインを簡単に整えられる  
  - Markdown を置くだけでページが生成される  
  - カスタマイズ性が高く、社内向けドキュメントサイトに最適

- **社内ドキュメント用途で Hugo が向いている理由**  
  - ローカルで完結して動く  
  - 外部サービス依存が少ない  
  - セキュリティ要件が厳しい環境でも運用しやすい  
  - GitHub と組み合わせると CI/CD で自動更新できる  
  - 長期的にメンテナンスしやすい
